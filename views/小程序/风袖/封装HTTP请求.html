          # 层与层调用时回调函数传递问题

          函数间调用，由于最里面一层函数的执行是异步的，就会导致层与层之间的调用都是异步

          # 异步调用的三种方式

          1.callback回调地狱，嵌套调用
          2.promise比callback强，解决回调地狱，但还是不够简洁
          promise的作用是将异步函数的执行结果保存到resolve、reject中，需要的时候调用then来处理异步函数执行的结果
          3.async await基于promise，只有一个函数返回promise，才可以使用async await

          使用await的前提是函数必须要返回一个结果，
          即使这个函数返回的是一个数值1 、'abc'等常量，也可以用await，但意义不大，
          因为await其实是求值，await+常量还是等于常量，没有意义，一般还是用在promise前，
          但小程序但wx.request或者其他但api不会返回任何值，更不会返回一个promise，
          小程序的云开发中，微信API，你不加success回调函数，那么它就会给你返回promise，就可以使用await，
          这里的wx开头的API中都不能直接返回promise，只能用回调函数方式接收调用结果。

          这里可以通过一个函数，封装wx的API，使它返回promise的结果， 然后就可以使用async await来处理异步调用

           const promisic = function (func) {
   return function (params = {}) {
     return new Promise((resolve, reject) => {
       const args = Object.assign(params, {
         success: (res) => {
           resolve(res);
         },
         fail: (error) => {
           reject(error);
         }
       });
       func(args);
     });
   };
 };

          # HTTP请求合并

          合并的目的在于减少数据库查询次数
          1.每周上新 归为Theme主题，但不同于顶部的主题，
          顶部主题只是一张图片，主题入口，
          这里不仅仅是主题入口，还需要把入口内部详情中的部分商品提取出来；

          2.之前theme.js的model中已经有了获取主题A的方法，是否还应该在写一次获取主题E的方法？
          从性能优化来讲，尽可能减少前端请求服务器的次数，
          整个home页面主题只应该发送一次请求，api提供来查询多条theme的传参方式；

          3.那为什么不把banner theme activity都写在一个接口中，让前端只发送一次请求就可以了呢？
          1）每一个数据都发送http请求；
          2）整个home只发送一个http；
          3）有选择的把部分http请求合并成一个；
          这三种方式的衡量方式：
          -http请求数量（性能指标）
          -每次http请求（完成相同数据请求）会引起多少次数据库的查询（性能指标）
          -接口的灵活性，接口的可维护性（非性能指标，人工）
          这里：
          第一种方式，http请求数量最多 查询数据库次数也是最多的 接口灵活性比较高，接口粒度最小，可维护性好；
          第二种方式，http请求数量最少 查询数据库次数小于第一次（不会有重复的查询数据库） 接口非常不灵活，维护成本最高，对于迭代开发来说是很不利的，可维护性越高 bug率越少；
          第三种方式，是比较中和的选择，考虑哪些http请求合并-参考依据是第二条指标，合并后会减少数据库查询次数；

          4.web应用的瓶颈在IO，IO密集型，redis、mysql读写；
          CPU密集型，视频的转码、编码；
          如果web应用中有CPU密集型，应该独立为一个服务，不应该放在web应用的服务器上，放在一个单独的服务器上；
